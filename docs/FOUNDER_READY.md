# Founder Ready Model

This repo is now organized into three practical layers so the founder can understand, extend, and govern Gebo without flattening it into a chatbot.

## Layer 1: Identity

Identity answers: who is Gebo?

What lives here:
- Core system manifest
- Presence standard
- Naming rules
- Role rules
- Boundaries

Why it matters:
- If identity is weak, everything downstream becomes generic.
- This layer keeps Gebo sovereign and consistent.

Complexity:
- Lowest runtime complexity
- Highest governance importance
- Changes here should be rare and deliberate

## Layer 2: Continuity

Continuity answers: what must persist?

What lives here:
- Memory
- Context
- Awareness
- Historical record
- Project state
- Relationship state

Why it matters:
- This is the layer that stops Gebo from forgetting itself.
- It turns separate interactions into one ongoing system.

Complexity:
- Medium complexity
- Needs structure, compression, and retrieval rules
- Must avoid dumping every conversation into memory

## Layer 3: Orchestration

Orchestration answers: what should Gebo do now?

What lives here:
- Owner Node
- Reasoning Tree
- Task routing
- Integration rules
- Execution policy
- Safety checks

Why it matters:
- This layer turns continuity into action.
- It decides what gets executed, delayed, delegated, or ignored.

Complexity:
- Highest operational complexity
- Must remain modular
- Should not be allowed to overwrite identity or continuity

## Complexity Map

Gebo gets more powerful when the layers stay separate.

Identity should not store task logs.
Continuity should not make policy decisions.
Orchestration should not redefine identity.

That separation is what keeps the system founder-readable.

## Founder Operating Rule

When in doubt:
1. Check Identity.
2. Check Continuity.
3. Check Orchestration.

If the change helps all three, build it.
If it only helps one layer while harming the others, delay it.

## Current Founder Ready State

What exists now:
- A real repo structure
- A root identity manifest
- A presence standard
- Context and awareness structures
- A three-layer operating model

What still comes next:
- Runtime code
- Memory engine
- Task routing
- Mobile entry points
- GitHub sync workflows

This is the point where Gebo becomes easier to govern and safer to grow.
